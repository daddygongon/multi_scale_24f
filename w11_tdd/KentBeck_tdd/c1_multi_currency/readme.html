1. Red, green, refactor (p. x)

Red
Write a little test that doesn't word, and perhaps doesn't even compile at first.
Green
Make the test work quickly, committing whatever sins necessary in the process.
Refactor
Eliminate all of the duplication created in merely getting the test to work.

Red/green/refacotor – the TDD mantra.

Courage. メタ心理学だね

2. rhythm of TDD (p.2)

  1. Quickly add a test.
  2. Run all test and see the new one fail.
  3. Make a little change.
  4. Run all tests and see them all succeed.
  5. Refactor to remove duplication.

3. The surprises (p.2)

  • How each test can cover a small increment of functionality
  • How small and ugly the changes can be to make the new tests run
  • How often the tests are run
  • How many teensy-weensy steps make up the refactroings
